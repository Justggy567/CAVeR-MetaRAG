# Construction Specification of the 700-Factoid Diagnostic Set

### 1. Overall Purpose

This project constructs a **700-factoid diagnostic set** designed to analyze the verification difficulty, scaling behavior, and small verifier failure patterns of different types of atomic facts in hallucination detection. The fundamental unit of analysis here is the **source factoid**, rather than sentences after synonym/antonym mutations. Each sample corresponds to an atomic factual proposition extracted from the original answer.

The diagnostic set was designed as a balanced factoid-level evaluation set. It contains seven factoid categories, and each category contains 50 context-supported factoids and 50 context-unsupported factoids. 

---

### 2. Source Datasets

The candidate pool for the 700-factoid diagnostic set is derived from two QA data sources:

1. **ASQA**
2. **HaluEvalQA**

The ASQA portion uses a fixed 96-example subset obtained from the previous stage.
The HaluEvalQA portion uses the cleaned HaluEvalQA file as input.

---

### 3. Candidate QA Sample Selection

#### ASQA

The ASQA script directly processes all samples in the input file. For the experiment, the input file is a fixed 96-example ASQA subset, meaning that the ASQA factoid pool is derived from the `right_answer` and `hallucinated_answer` fields of these 96 samples.

#### HaluEvalQA

The HaluEvalQA script first reads the cleaned HaluEvalQA data.
The HaluEvalQA factoid pool is then obtained from this cleaned HaluEvalQA data.

---

### 4. Factoid Extraction

For each QA sample, the script processes both answer fields simultaneously:

```python
answers = [
    ("right", item.get("right_answer", "")),
    ("hallucinated", item.get("hallucinated_answer", ""))
]
```

This means candidate factoids are derived from both:

1. `right_answer`
2. `hallucinated_answer`

Subsequently, the script calls the following method from the MetaRAG framework:

```python
detector.decompose_answer(answer_text)
```

This splits the complete answer into several atomic factoids. Empty strings or factoids containing only whitespace are filtered out. Both the ASQA and HaluEvalQA construction scripts apply this logic.

**Important Note:**
`answer_type = right` does not automatically imply `context_support = True`;
`answer_type = hallucinated` does not automatically imply `context_support = False`.

The script invokes the oracle verifier for each extracted factoid to independently assess whether it is supported by the context. Therefore, the final `context_support` label is a factoid-level oracle judgment rather than a direct inheritance of the original answer's label.

---

### 5. Factoid Taxonomy and Classification Rules

Each factoid is assigned to one of seven categories.

The classification is based on rule-based keyword / regex matching using a fixed priority order:

```text
Cat 2 > Cat 5 > Cat 3 > Cat 7 > Cat 4 > Cat 6 > Cat 1
```

The specific categories are as follows:

| Category | Explanation | Main Signal |
| :--- | :--- | :--- |
| **Cat 1: Entity/Alias/Lexical Rewrite** | Entities, aliases, and lexical rewrites | Fallback category |
| **Cat 2: Negation/Exception** | Negation and exceptions | *not, never, without, except, none*, etc. |
| **Cat 3: Quantifier/Boundary** | Quantifiers, scopes, and boundaries | *only, all, every, some, majority*, etc. |
| **Cat 4: Time/Sequence** | Time and sequences | Years, months, *before, after, since, first, last*, etc. |
| **Cat 5: Comparative/Superlative** | Comparatives and superlatives | *more than, higher, lower, most, least*, etc. |
| **Cat 6: Numerical/Computation** | Numerical values, units, and computation | Numbers, *percent, average, total, double*, etc. |
| **Cat 7: Multi-entity/Multi-hop** | Multi-entity relations, multi-hop reasoning, and causal relationships | *because, due to, acquired, founded by, part of*, etc. |

The classifier first checks for Negations/Exceptions (Cat 2) because the core risk of these factoids is logical reversal. It then evaluates Comparatives/Superlatives (Cat 5), as comparative structures often involve numerical elements but rely on distinct error patterns. Time/Sequence (Cat 4) is placed before Numerical/Computation (Cat 6) to prevent years or eras from being misclassified under generic numerical values. If a factoid does not meet the criteria of the first six categories, it falls back to Cat 1 (basic entities, aliases, or lexical rewrites).

---

### 6. Oracle Support Labeling

The `context_support` label for each factoid is generated by the DeepSeek oracle verifier. The script initializes the detector using `MetaRAGDetector(model="deepseek-chat", use_local_verifier=False)` and relies solely on DeepSeek for factoid-level oracle judgments.

The oracle judgment rules are as follows:

* **Output `YES`**: The factoid is explicitly stated in the context, or can be strictly logically entailed by the context.
* **Output `NO`**: The factoid contradicts the context, or the context lacks sufficient information to verify the factoid.
* External knowledge is not allowed.
* The model must output only `YES` or `NO`.

When parsing the model's output, the script only accepts responses starting with `YES` or `NO`. If there is no response, an output exception, or a failure to match `YES` / `NO`, the label defaults to `False` (i.e., not supported by the context).

This step ensures that the final diagnostic set labels are based on the actual factoid-level support relation relative to the context, rather than simply inheriting the original answer labels.

---

### 7. Intermediate Factoid Record Format

In the ASQA construction script, each factoid record contains the following fields:

| Field | Meaning |
| :--- | :--- |
| `id` | Globally incremented ID |
| `source_id` | Original QA sample ID |
| `fact_id` | Composed of `source_id`, `answer_type`, and the local factoid index |
| `context` | Evidence context used for verification |
| `question` | Original question |
| `factoid` | Extracted atomic factoid |
| `factoid_type` | One of the seven taxonomy labels |
| `context_support` | True / False label based on the oracle judgment |
| `answer_type` | Source of the factoid: `right` or `hallucinated` |
| `source_answer` | Original answer text |

The schema of the fields in the HaluEvalQA construction script is identical.

---

### 8. Merging Candidate Pools

After ASQA and HaluEvalQA generate their respective factoid-level candidate files, these files are merged into a single comprehensive candidate pool.

The final 700-factoid diagnostic set is not generated directly from the raw QA data, but is sampled from this constructed factoid candidate pool. Each candidate in the pool must contain at least the following fields:

```text
factoid_type
context_support
```

This requirement exists because the final balancing script performs bucketing and stratified sampling based on these two attributes.

---

### 9. Balanced Sampling to 700 Records

The final script `balanced_factoid_to_700.py` first creates buckets based on:

```text
factoid_type × context_support
```

Represented as:

```python
buckets[factoid_type][context_support]
```

Subsequently, the script extracts the following for each factoid category:

```text
50 supported factoids
50 unsupported factoids
```

This results in 100 records per category, bringing the total across all seven categories to 700. The script lists the seven categories and samples from the `True` and `False` pools for each.

It then reassigns the globally incremented ID:

```python
item["id"] = idx
```

Finally, the dataset is saved as:

```text
balanced_factoid_diagnostic_dataset_700.json
```

---

### 10. Manual Filtering and Replacement

The resulting 700-item dataset underwent manual review to assess the validity and logic of each entry. Any identified unreasonable or problematic records were revised or replaced by drawing suitable substitutes from the candidate pool.
